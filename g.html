1. Data

Definition (As per this course)

  • Data is any fact that can be recorded and stored.
  • If something can be recorded, it is considered data.

Examples of Data

  • Text ‚Äì name, address, description
  • Numbers ‚Äì phone number, marks, age
  • Images ‚Äì photographs, satellite images
  • Audio ‚Äì speech, songs
  • Video ‚Äì movies, recorded lectures
Core idea: Any recordable fact is considered data.

2. Database

Definition

  • A Database is a collection of related data.
  • Data stored must be related to a specific application or purpose.
  • Only relevant data is inserted, deleted, or updated.

Examples

  • Student database ‚Üí name, roll number, marks, address.
  • Ticket booking database ‚Üí passenger details, ticket ID.

3. Types of Databases

3.1 Traditional Database

  • Stores text and numbers only.
  • Examples: Student records, banking systems.

3.2 Multimedia Database

  • Stores images, audio, video.
  • Examples:
    • YouTube, Vimeo ‚Üí video databases.
    • Music platforms ‚Üí audio databases.

3.3 Geographic Information System (GIS)

  • Stores image‚Äëbased and spatial data.
  • Examples:
    • Satellite images of Earth.
    • Space images/videos from NASA.

3.4 Real‚ÄëTime Database

  • Data must be updated immediately.
  • Used where instant response is required.
  • Examples:
    • Supermarket inventory.
    • Production and manufacturing systems.

3.5 Data Warehouse

Definition

  • A Data Warehouse is a database that stores:
    • Very large volume of data.
    • Historical data (long‚Äëterm past data).

Examples

  • Stock market data of 50‚Äì100 years.
  • Company sales data over decades.
Core idea: A data warehouse is used mainly for analysis and prediction, not daily transactions.

4. Manual vs Computerized Databases

4.1 Manual Databases (Earlier)

  • Database ‚Üí registers/books.
  • Manipulation ‚Üí pen and paper.

4.2 Computerized Databases (Now)

  • Data stored on computers.
  • Stored on hard disks / storage devices.
  • Requires software to manage data.

5. Need for Database Software

To manage a database, software must support:

  1. Define
    • Data types (int, char, string, etc.).
    • Structure of data.
  2. Construct
    • Store data physically on storage devices.
  3. Manipulate
    • Insert data.
    • Delete data.
    • Update data.
    • Retrieve data.

6. Why Not Use Only the Operating System?

Using the OS directly for databases causes problems:

  • Increases OS size and complexity.
  • Slower access due to user mode ‚Üí kernel mode switching.
  • OS provides unnecessary services for database access.
Core idea: Databases require fast access with minimal overhead.

7. Database Management System (DBMS)

Definition

  • A DBMS is a set of programs used to:
    • Define the database.
    • Construct the database.
    • Manipulate the database.

Note: DBMS is software, not data.

Simple Analogy

  • Database ‚Üí Book.
  • DBMS ‚Üí Pen.

8. Database System (DBS)

Definition

  • A Database System (DBS) is: Database + DBMS.

Meaning

  • Database ‚Üí Collection of related data.
  • DBMS ‚Üí Programs to manage the data.
  • Together they form a Database System.

2. Levels of Data Modeling in Databases

Similar to house construction, database design uses three levels:

  • Conceptual (High-Level) Model
  • Representational (Logical) Model
  • Physical (Low-Level) Model

3. Conceptual Model (High-Level Model)

Purpose

  • Used to explain database design to non-technical users.
  • Focuses on what data is stored, not how it is stored.

Example Scenario

  • Database designer discusses requirements with:
    • School principal
    • Manager
    • Client (no DBMS knowledge)
Important: No technical terms like SQL, tables, records are used.

Conceptual Modeling Technique

The most popular conceptual model is:

  • Entity‚ÄìRelationship (ER) Model

Components

  • Entities ‚Äì Real-world objects
    Example: Student, Course
  • Attributes ‚Äì Properties of entities
    Student ‚Üí Name, Roll No, Age
  • Relationships ‚Äì Association between entities
    Student attends Course

Benefits: ER diagrams make database ideas easy to visualize and understand.

‚úîÔ∏è ER Model is included in GATE syllabus

4. Representational Model (Logical Model)

Purpose

  • Used by database designers and programmers.
  • Shows how data is logically organized.

Key Characteristics

  • Uses tables (relations).
  • Rows ‚Üí Records.
  • Columns ‚Üí Attributes.

Name of the Model: Relational Model

Key: Bridges the gap between conceptual design and physical storage.

5. Physical Model (Low-Level Model)

Purpose

  • Describes how data is actually stored in memory or disk.
  • Concerned with efficiency and storage details.

Key Details Covered

  • Record structure.
  • Data types.
  • Byte-level storage.
  • How records are accessed.

Example

  • Table is not stored as an image.
  • Data is stored as a sequence of bytes.
  • To access the 3rd record, system must skip bytes of previous records.
Key: Focuses on storage and access mechanisms.

6. Order of Database Design

Database design proceeds in this sequential order:

1

3

Final: When all three levels are completed, database design is finished.

1. ER Model and Software Design Analogy

In software design, UML diagrams (class diagrams, etc.) are used to represent structure before coding.

In database design, ER diagrams play a similar role at the conceptual level.

Key: ER diagrams graphically represent database design before implementation.

2. Purpose of ER Diagrams

ER diagrams are used to:

  • Capture user requirements expressed in words.
  • Convert textual requirements into pictorial form.

They improve communication between:

  • Database designer (technical).
  • End users (non‚Äëtechnical).

Users describe what they want in natural language; the designer converts this into ER diagrams.

3. Core Components of ER Model

The ER model consists of three main components:

Entity ‚Äì Real‚Äëworld object.
Attribute ‚Äì Property of an entity.
Relationship ‚Äì Association among entities.

4. Entity

Definition: An Entity is a real‚Äëworld object that can be uniquely identified and stored in a database.

Examples:

  • Person
  • Student
  • Employee
  • Car
  • Department
Note: Any object we want to record information about becomes an entity.

5. Attributes

Definition: Attributes are the properties used to describe an entity.

If Person is an entity, its attributes may include:

  • Name
  • Age
  • Address
  • Phone number
Key: Attributes provide detailed description of entities.

6. Relationship

Definition: A Relationship represents an association among entities.

Relationships are usually expressed using verbs.

Examples:

  • Employee works for Department
  • Student attends Course
  • Person owns Car
Note: Relationships describe how entities are connected.

7. Entity Type vs Entity Instance

Entity Type
  • Name of the entity.
  • Example: STUDENT
  • Also called Intension.
Entity Instance
  • Actual data records.
  • Example: (RollNo = 101, Name = Ravi).
  • Also called Extension.
Important: In ER diagrams, we usually show entity types, not all instances.

8. Relationship Type vs Relationship Instance

Relationship Type ‚Üí Name of the relationship.

Example: WORKS_FOR

Relationship Instance ‚Üí Actual association.

Example: Ravi works for CSE Dept.

Idea: Same concept as entity type vs instance applies to relationships.

9. Why ER Diagrams Are Abstract

  • ER diagrams do not show all data values.
  • They show only entity types, attributes, and relationships.
  • This keeps the diagram simple and understandable.
Goal: Clarity, not completeness of every data value.

10. ER Modeling as an Iterative Process

ER modeling is a cyclic and interactive process:

  1. Collect requirements from users.
  2. Draw initial ER diagram.
  3. Show diagram to users.
  4. Refine based on feedback.
  5. Repeat until requirements are clear.

1. What are Attributes?

Attributes are properties used to describe entities or relationships.

The more attributes an entity has, the better it is described, which improves data clarity and query usefulness.

Example (Entity: Person)

  • Name
  • Age
  • Address
  • Phone Number

Attributes help answer queries like students by city or employees older than 30.

2. Classification of Attributes

Attributes are classified in multiple independent ways. These classifications are not mutually exclusive.

One attribute can belong to multiple categories at the same time.

3. Simple vs Composite Attributes

3.1 Simple Attribute

  • Cannot be divided further.
  • Atomic in nature.

Examples:

  • Age
  • Salary
  • First Name

3.2 Composite Attribute

  • Can be divided into sub‚Äëparts.
  • Made up of multiple simple attributes.

Examples:

  • Name ‚Üí First Name, Middle Name, Last Name
  • Address ‚Üí Street, City, State, PIN
Note: Name is a composite attribute, while First Name is a simple attribute.

4. Single‚ÄëValued vs Multi‚ÄëValued Attributes

4.1 Single‚ÄëValued Attribute

  • Has only one value at any time.

Examples:

  • Age
  • Date of Birth

4.2 Multi‚ÄëValued Attribute

  • Can have multiple values at the same time.

Examples:

  • Phone Numbers (mobile, landline)
  • Addresses (home, office)
Important: Age can never be multi‚Äëvalued; phone number can be.

5. Stored vs Derived Attributes

5.1 Stored Attribute

  • Value is stored directly in the database.

Examples:

  • Date of Birth
  • Salary

5.2 Derived Attribute

  • Value is computed from other attributes.

Examples:

  • Age (derived from Date of Birth)
  • Total Marks (derived from subject‚Äëwise marks)
Note: Derived attributes are usually not stored to avoid redundancy.

6. Null Values (Important Concept)

A NULL value is used when:

  • Value does not exist (e.g., phone number not available).
  • Value is not applicable (e.g., no middle name).
Very Important: NULL ‚â† 0 ‚â† blank.

7. Complex Attribute

A Complex Attribute is:

  • Composite and
  • Multi‚ÄëValued

Examples:

  • Address (multiple addresses, each having street, city, PIN)
  • Phone Numbers (multiple numbers, each with STD code + number)
Formula: Complex = Composite + Multi‚ÄëValued.

8. Important Exam Insight

An attribute can belong to multiple categories simultaneously:

Attribute Type Combination
Name Composite + Single‚ÄëValued
Age Simple + Single‚ÄëValued + Derived
Address Composite + Multi‚ÄëValued (Complex)
Date of Birth Simple + Single‚ÄëValued + Stored
GATE / EXAM TIP

Always check:

  • Can it be split? ‚Üí Composite vs Simple.
  • Can it have many values? ‚Üí Multi‚ÄëValued vs Single‚ÄëValued.
  • Is it computed? ‚Üí Derived vs Stored.

9. GATE / Exam‚ÄëType Question Pattern

Questions usually give a verbal description and ask: "What type of attribute should be used?"

Examples:

  • Employee can reside in two cities ‚Üí Address = Multi‚ÄëValued.
  • Address has street, city, PIN ‚Üí Address = Composite.
  • Employee has two addresses with full details ‚Üí Address = Complex.
  • Employee has mobile and landline ‚Üí Phone Number = Multi‚ÄëValued.
  • Name has first and last name ‚Üí Name = Composite.
FAST RECALL

Complex = composite structure + multiple values per entity.
Think: "many detailed entries" ‚Üí Complex attribute.
